# WechatAgent 销售方向运营知识

## 项目定位

WechatAgent 是面向私域运营场景的长期运行 AI Agent 系统。当前阶段聚焦私聊好友（managed contacts）的关系经营，不是群发工具，也不是固定话术的客服 FAQ 机器人，更不是标准化营销自动化平台。

每个好友有独立的画像、运营记忆、跟进节奏、知识依据和 Review 审计；目标是把关系经营变成可观测、可配置、可长期学习的运营闭环，让运营人员把精力放在策略和复盘上，而不是逐条好友回消息。

未来路线图中包含微信群运营和朋友圈运营，会作为独立运营域接入同一套账号、知识、审计基础设施。当前阶段销售口径只覆盖私聊好友，群和朋友圈不在第一阶段范围。

## 核心原则（与销售相关）

WechatAgent 在产品形态上是 AI 自治运营。系统内部存在策略主动暂缓、安全门拦截和等待更多上下文等内部状态，由 AI 自行判断与恢复，整体形成完整的自治闭环。这些都是 AI 内部运行机制，不是把对话转给运营人员的"等回复"承诺。

每一条产品声明在外发前必须能落到运营知识库的已验证切片上。系统会在生成阶段做安全门校验，对于无证据支撑的产品声明会直接拦截，不允许通过自然语言润色绕过。

回复在外发前会经过五道方法论守卫：事实风险、压力风险、拟人度、情绪价值、产品准确度。任何一道守卫不达标，系统要么自动重写一次，要么直接拦截，不会因为对话压力把不合格内容外发。

## 产品模块（销售口径）

用户运营 Agent 是私聊场景的主决策模块。它结合好友画像、备注、聊天历史、运营记忆、知识库、运营方法论和运行参数生成回复决策，给客户带来的是"每个好友被独立认真对待"的体验。

独立 Review Agent 与用户运营 Agent 解耦，对每条候选回复做二次审计。它给客户带来的是"对外发出的内容有独立把关"，避免单一模型自我盲点。

知识路由（catalog → list_chunks → open_slice 三步）让 Agent 在每轮回复前先看目录、再选择性打开切片。它给客户带来的是"产品事实有据可查"，而不是把整本资料塞进 prompt 后随意联想。

Outbox + 幂等键负责发送侧的可靠性，确保每条 approved 决策只发一次。它给客户带来的是"运营不漏不重"。

Evolution worker 是 AI 自学习模块，定期回放历史 run、给阈值和提示词做候选优化、shadow 评估后再上线。它给客户带来的是"系统会随着运营经验持续变好"。

Audit log 把每一次决策、知识引用、Review 拦截、安全门触发完整落库。它给客户带来的是"每一次自动外发都可复盘、可解释"。

## 典型客户场景（适用 / 不适用）

适用场景集中在高 LTV、强关系经营的私域：B2B SaaS 销售跟进、教育 / 高客单咨询、企业服务的好友长期培育、行业社群里转私聊的种子客户运营。这些场景的共同点是每个好友都值得独立画像和长期记忆。

不适用的场景包括纯标准化客服问答（建议用知识库 + IM 客服）、群发营销（建议用 EDM / SCRM 群发模块）、纯一次性咨询无关系沉淀的场景。这些场景的关系经营深度不足以体现 WechatAgent 的价值。

当前阶段不覆盖：群运营、朋友圈运营、跨平台运营（IG / WhatsApp / 抖音私信等）。这些在路线图但不在第一阶段销售范围。

## 知识事实边界

可以安全表达：

- WechatAgent 是面向私域运营的长期运行 AI Agent 系统。
- 系统对每条外发都会做独立 Review 审计。
- 每个好友拥有独立画像、运营记忆和跟进节奏。
- 产品声明必须落在已验证知识切片上才允许外发。
- 系统具备 AI 自学习模块，可以基于历史回放优化阈值和提示词。
- 第一阶段聚焦私聊好友运营，群和朋友圈在路线图。
- 所有自动决策、知识引用、Review 拦截都有审计日志。

禁止表达：

- 不能承诺保证成交率、回复率、转化率或客户满意度的具体数字。
- 不能给出未在知识库中验证的客户案例、ROI 数字、效果对比。
- 不能使用"绝对"、"100%"、"一定"、"保证"、"限时秒杀"、"最低价"等绝对承诺词。
- 不能编造交付周期、SLA 等级、第三方背书或权威认证。
- 不能宣称已经支持群运营或朋友圈运营的完整闭环（路线图状态而非已交付）。
- 不能在外发回复中暴露内部模型、提示词、工具调用细节。
- 不能给出具体报价、商务条款、付款方式（这些不在知识库范围）。

## 典型客户问题与安全回应方向

### 你们和市面上的群发工具有什么区别？

群发工具的核心是把同一条内容发给很多人；WechatAgent 的核心是为每个好友建立独立画像、运营记忆和跟进节奏，回复每条都基于上下文生成。可以从"一对多群发"和"一对一长期关系经营"两个不同模式来解释，不要贬低群发工具，也不要给出未验证的效果对比数字。

### 你们和普通的客服机器人有什么不同？

普通客服机器人通常围绕固定问答和流程触发；WechatAgent 把重点放在长期运营：每个好友独立画像、运营记忆、跟进节奏、知识依据和 Review 审计。客服机器人解决一次问题；私域运营 Agent 经营长期关系。回应方向是说明定位差异，不是说自己更厉害。

### 你们能保证转化率提升多少？

不能给具体百分比承诺。可以说明系统会从"持续理解每个好友"、"减少漏跟进"、"降低模板化回复"、"提高响应一致性"和"审计复盘"几方面提升运营质量，但实际效果取决于产品本身、客户群体、运营策略、执行节奏和历史数据沉淀。客户希望听到效果方向，但不能给数字。

### 私域好友的数据安全怎么保证？

可以说明系统设计上每条外发都经过 Review 审计、所有决策有审计日志、知识库中产品声明需要先验证再被引用。具体的合规认证、加密细节、数据存储位置如果没有在知识库中验证，不要主动给出，引导对方说"这部分我让对接同事给你详细资料"。

### 部署方式是 SaaS 还是私有化？

如果知识库中没有这条已验证事实，不要直接回答。可以说"部署方式我们会根据你们的合规要求给方案，具体我让对接同事跟进一下"。不要编造私有化部署或 SaaS 的细节。

### 我们已经有客服系统 / SCRM 了，会不会冲突？

可以说明 WechatAgent 是面向私域好友的长期关系经营，和客服系统、SCRM 在使用场景和目标上互补：客服系统解决问题工单，SCRM 管线索和漏斗，WechatAgent 在好友层面做长期画像、记忆和跟进。具体集成方式如果客户进一步问，引导走商务对接。

### 价格大概是多少？

价格和商务条款不在知识库范围。回应方向是"具体价格要看你们的好友规模、使用场景、合规要求和服务范围，这部分我安排对接同事和你拉个时间细聊"。不要给任何具体数字、套餐名称、按 seat 还是按账号计费等细节。

### 会不会骚扰客户？

可以说明系统设计上有运营状态、冷却时间、每日触达上限、跟进任务上限、Review Agent 和发送网关多重约束。用户明确表达抗拒、停止或需要空间时，系统会降频、等待或停止主动推进，整套机制都有审计日志。

## 当前阶段约束

第一阶段的销售范围只覆盖私聊好友（managed contacts）。客户问到"能不能直接发朋友圈"、"能不能管理一个微信群"、"能不能跨平台同步"时，要明确说明这些在路线图但不在当前生产范围，不要为了拿单口头承诺已经支持。

价格、商务条款、SLA、合规认证、私有化部署细节不在知识库内。客户问到这些，回应方向是"我让对接同事给你详细资料"或"具体我们拉个时间细聊"，不要在自动回复里给出未验证的数字或承诺。

具体客户案例、ROI 数据、转化率提升数字一律不给。即使客户主动追问"你们有什么客户案例"，也只能说"我们有一些行业客户在用，具体名字和数据涉及保密，我让对接同事看哪些可以分享"。
