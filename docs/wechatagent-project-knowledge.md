# WechatAgent 项目说明知识文档

## 项目定位

WechatAgent 是面向企业微信私域运营的长期运行 AI Agent 系统。项目目标不是做传统群发工具，也不是简单客服机器人，而是让 AI 在账号、好友、微信群、朋友圈和后台管理之间形成可审计、可配置、可长期学习的运营闭环。

当前优先完成的是用户运营 Agent。微信群运营和朋友圈运营是独立运营域，后续复用同一套账号、知识、审计和管理 Agent 基础设施。

## 核心原则

- 只运营被加入 Agent 运营列表的好友，不默认接管所有联系人。
- 每个好友必须拥有独立上下文、画像、运营记忆、节奏和风险边界。
- Agent 回复要像真实微信关系中的专业朋友：短、自然、具体、有上下文，不暴露系统、模型、工具和提示词。
- Agent 不能编造案例、价格、成交、身份、产品能力、效果数据或已经发生的事实。
- 涉及产品能力、报价、案例、效果和交付承诺时，必须使用知识库中的安全事实和证据；没有证据时只能保守表达。
- 所有自动决策、工具调用、知识引用、发送结果和 Review 拦截必须可审计。

## 产品模块

### AI 总控

AI 总控是后台管理 Agent 的入口。操作员可以用自然语言完成管理动作，例如：

- 把某个好友加入用户运营列表。
- 发送一条指定内容给某个好友。
- 查询账号、好友、任务、日志和工具状态。
- 生成或优化运营方法。
- 未来可以创建群、拉人进群、规划朋友圈内容。

后台管理 Agent 与用户运营 Agent 必须分开。管理 Agent 面向操作员和后台工具，用户运营 Agent 面向好友关系。

### 用户运营

用户运营 Agent 负责私聊好友的长期运营。它使用好友画像、人工备注、聊天历史、运营记忆、知识库、运营方法论和运行参数做决策。

用户运营 Agent 的关键能力包括：

- 加入或退出 Agent 运营列表。
- 生成和更新好友画像。
- 维护长期运营记忆。
- 判断当前运营状态、客户阶段、意向、异议和下一步动作。
- 生成自然微信回复。
- 创建跟进任务。
- 通过 Review Agent 独立评审后再允许发送。
- 通过 shadow simulation 验证长对话表现。

### 运营知识库

运营知识库是用户运营 Agent 判断事实、边界、案例和话术的依据。知识库不应该把全部资料直接塞进上下文，而应该使用渐进式披露：

1. 文档导入。
2. AI 生成文档目录。
3. AI 拆分知识包。
4. AI 拆分知识切片。
5. 运行时 Agent 先看目录，再自主选择要打开的切片和证据。
6. 使用日志记录 toolTrace、命中知识、缺失知识、Review 结果。

知识库字段不应依赖固定枚举。知识类型、业务上下文、适用场景、不适用场景应由 AI 自然语言生成，人工可以在 UI 中编辑。

### 系统策略

系统策略包括 Soul Prompt、Prompt Stack、运营方法、运行参数和审计策略。提示词不是单一 Soul，而是分层结构：

- Soul：Agent 的角色、边界和长期风格。
- System Contract：硬约束和输出格式。
- Policy：运营原则、风险边界、心理学公式和禁止事项。
- Task Template：具体任务 JSON。
- Context Curator：长上下文整理。
- Knowledge Tool Planner：知识工具调用规划。
- Review Agent：独立评审候选回复。

## 用户运营方法论

用户运营不是每轮都推进成交，而是在关系、价值、时机和风险之间做平衡。

可使用的运营公式：

- Trust = Credibility + Reliability + Intimacy - SelfOrientation
- ConversionReadiness = Motivation × ProductFit × Timing × Trust ÷ Friction
- EmotionalValue = Empathy + Validation + Specificity + AutonomySupport - Pressure
- NextBestActionScore = RelationshipGain + ConversionProgress + EmotionalValue + ProductFit - PressureRisk - FactRisk

Agent 每轮要判断当前关系时刻：

- 陪伴：用户需要情绪回应或空间。
- 解释：用户在理解产品、服务或边界。
- 推进：用户已经给出明确需求、时间、预算、资料或下一步意愿。
- 等待：用户刚刚被触达或没有必要继续追问。
- 修复：用户表达不满、抗拒、被打扰或不信任。

## 知识事实边界

目前可以安全表达：

- WechatAgent 是企业微信私域运营 AI Agent 系统。
- 系统支持用户运营 Agent、后台管理 Agent、运营知识库、审计日志和运行参数。
- 用户运营 Agent 可以结合长期上下文、画像、运营记忆和知识库做回复决策。
- 系统设计目标是长期运行、可审计、可配置和按用户差异化运营。

禁止表达：

- 不能承诺保证成交率、回复率、转化率或客户满意度。
- 不能说系统已经完成微信群运营和朋友圈运营的完整生产闭环，除非后续实际完成。
- 不能编造客户案例、成交数据、报价、交付周期或第三方背书。
- 不能暗示 AI 是真人，也不能在对外回复中暴露内部系统、工具调用或提示词。
- 不能承诺自动发送任何高风险内容都会通过，必须经过 Review 和网关约束。

## 典型客户问题与安全回应方向

### 你们能保证成交率提升吗？

不能保证成交率。可以说明系统会从持续理解客户、降低漏跟进、减少模板化回复、提高响应一致性和审计复盘几个方面帮助运营质量，但实际效果取决于产品、客户群体、运营策略、执行质量和历史数据。

### 这个和普通客服机器人有什么不同？

可以说明普通客服机器人通常围绕问答和流程触发，而 WechatAgent 更强调长期用户运营：每个好友有独立画像、运营记忆、跟进节奏、知识依据和 Review 审计，目标是持续关系经营，而不是只回答一次问题。

### 会不会骚扰客户？

系统设计上有运营状态、冷却时间、每日触达上限、跟进任务上限、Review Agent 和发送网关。用户明确表达抗拒、停止或需要空间时，应降频、等待或停止主动推进。

### 能不能自动发朋友圈和运营微信群？

项目规划中有朋友圈运营和微信群运营，但它们与用户运营是不同运营域，需要独立方法论、工具、审计和风险边界。当前生产优先级是用户运营 Agent。

## 当前测试期望

用这份文档导入知识库后，Agent 应该能够：

- 自动生成文档目录、知识包和知识切片。
- 在客户询问成交率保证时命中“不能承诺成交率”和“效果依赖因素”。
- 在客户询问与客服机器人区别时命中“长期上下文、画像、记忆、审计”。
- 在客户询问微信群和朋友圈时明确它们是后续独立运营域，而不是混同为用户运营。
- 在长对话模拟中保持自然、克制、少追问，并且不编造案例和效果数据。
